# 云端部署

这份文档记录 `ZJU-mentor` 的 Cloudflare 上线、数据库初始化、后台口令、自动更新和后续代码更新流程。

## 当前推荐方案

项目上线使用：

- `Cloudflare Workers`：运行后端 API，同时托管前端静态资源
- `Cloudflare D1`：云端 SQLite 数据库
- `Cloudflare Browser Rendering / Browser Run`：云端爬虫
- `Cron Triggers`：自动定时更新数据库
- `zjumentors.com`：Worker Custom Domain

当前代码里已经配置了两个自定义域名：

```json
"routes": [
  {
    "pattern": "zjumentors.com",
    "custom_domain": true
  },
  {
    "pattern": "www.zjumentors.com",
    "custom_domain": true
  }
]
```

## 需要购买什么

如果只是先上线，可以先用 Cloudflare 免费额度：

- Workers Free：可部署 Worker 和静态资源
- D1 Free：每天 `5 million` rows read、`100,000` rows written，总存储 `5 GB`
- Browser Rendering Free：每天 `10 minutes` browser time

这个项目初期的网页访问和 D1 数据量很小，D1 免费额度基本够用。真正紧张的是 Browser Rendering 的免费额度，因为全量抓取全校导师本地实测大约需要 20 分钟。现在代码已经改成定时任务每天只更新一个学部，尽量贴合每天 10 分钟免费额度。

如果后续发现每日自动爬虫经常因为 `429 Browser time limit exceeded for today` 失败，再升级 `Workers Paid`。Cloudflare 官方当前 Workers Paid 是每个账号最低 `$5/month`，Browser Rendering Paid 包含 `10 hours/month`，超出后按 browser hour 计费。

参考：

- Cloudflare Workers Pricing: https://developers.cloudflare.com/workers/platform/pricing/
- Cloudflare D1 Pricing: https://developers.cloudflare.com/d1/platform/pricing/
- Browser Rendering Limits: https://developers.cloudflare.com/browser-rendering/limits/
- Browser Rendering Pricing: https://developers.cloudflare.com/browser-rendering/platform/pricing/

## 设置后台口令

生产环境后台口令使用 Cloudflare Secret，不写进仓库：

```bash
npx wrangler secret put ADMIN_TOKEN
```

输入一个长随机字符串。以后访问后台登录页时使用这个字符串：

```text
https://zjumentors.com/__admin__/login
```

如果忘记后台口令，重新执行 `npx wrangler secret put ADMIN_TOKEN` 覆盖即可。

本地开发口令在：

```text
/Users/pipi/Life/ZJU-mentor/cloudflare-backend/.dev.vars
```

`.dev.vars` 已被 `.gitignore` 忽略，不应该提交。

## 初始化远程 D1

当前初始化使用仓库里的 canonical JSON：

```text
/Users/pipi/Life/ZJU-mentor/data/zju_colleges.json
/Users/pipi/Life/ZJU-mentor/data/all_zju_teachers.json
```

生成并导入远程 D1：

```bash
cd /Users/pipi/Life/ZJU-mentor/cloudflare-backend
npm run d1:seed:remote
```

这个命令会：

- 读取 `data/zju_colleges.json`
- 读取 `data/all_zju_teachers.json`
- 读取 `cloudflare-backend/migrations/0001_init.sql`
- 生成 `cloudflare-backend/tmp/seed.sql`
- 把 schema 和初始数据导入 Cloudflare D1

`tmp/seed.sql` 是生成物，可以随时重新生成，不提交。

## 自动更新会每天跑吗

会。`cloudflare-backend/wrangler.jsonc` 里配置了 Cron：

```json
"triggers": {
  "crons": [
    "0 19 * * *"
  ]
}
```

Cloudflare Cron 使用 UTC 时间。`0 19 * * *` 对应北京时间每天 03:00。

现在代码里的 `scheduled()` 不再每天全量更新，而是每天更新一个学部。分组逻辑是：

- 人文学部
- 社会科学学部
- 理学部
- 工学部
- 信息学部
- 农业生命环境学部
- 医药学部
- 国际校区
- 其他单位

每天会根据日期轮换一个分组。`其他单位` 也算一个分组。

注意：`其他单位` 里的单位数量较多，可能仍然接近或超过 Browser Rendering 免费每天 10 分钟额度。如果这天失败，可以手动补跑，或者升级 Workers Paid。

## 手动更新数据库

本地手动触发：

```bash
curl -s -X POST http://127.0.0.1:8787/api/admin/sync/run \
  -H 'X-Admin-Token: local-dev-admin-token'
```

线上手动触发：

```bash
curl -s -X POST https://zjumentors.com/api/admin/sync/run \
  -H 'X-Admin-Token: your-production-admin-token'
```

当前这个手动接口跑的是全量同步，会比定时任务更耗 Browser Rendering 时间。免费计划下不要频繁手动全量跑。

查看同步记录：

```bash
curl -s https://zjumentors.com/api/admin/sync/runs \
  -H 'X-Admin-Token: your-production-admin-token'
```

同步记录在 D1 的 `sync_runs` 表中。

## GitHub Release 放 db 文件的问题

不建议把运行时数据库放到 GitHub Release。原因：

- 网站运行时已经使用 Cloudflare D1，不需要下载 db 文件
- D1 存储免费额度当前有 `5 GB`，这个项目远小于这个量级
- GitHub Release 更适合发布备份文件，不适合作为线上数据库
- 如果用户从 GitHub Release 下载文件，流量主要走 GitHub，不走 Cloudflare Workers
- 但 GitHub 本身也有服务条款和滥用限制，不适合当数据库 CDN

更稳的方案：

- 线上运行用 D1
- 定期从 D1 导出备份 SQL 或 JSON
- 备份可以放 GitHub Release、Cloudflare R2、本地硬盘三选一

如果以后备份文件很大，放 Cloudflare R2 会更像正规备份方案，但 R2 的 Class A/B 操作和存储会按 R2 规则计费。

## 成本判断

当前最可能产生费用的是 Browser Rendering，不是 D1，也不是静态网页流量。

短期建议：

- 先用免费计划上线
- 观察 `sync_runs` 里每天的定时任务是否成功
- 如果 `其他单位` 那天经常失败，再考虑升级 Workers Paid
- 如果访问量变大，再看 D1 Metrics 和 Workers Metrics

Cloudflare Workers Paid 官方说明包含无额外数据传输费。D1 主要按 rows read、rows written 和超额存储计费。
